## Step 1: Exclude `src/driver/` from Main Project Checks

**Why:** The vendored Cypress driver code was never written for our
strict tsconfig or eslint rules. Including it in the main project
checks causes 2,405 type errors and 836 lint problems, all from
`src/driver/`. This blocks every PR check command in `AGENTS.md`.

### 1.1 Exclude from `tsconfig.json`

Add `"src/driver/**"` to the `exclude` array in the root
`tsconfig.json`. The driver already has its own `src/driver/tsconfig.json`.

**Verify after:**

```bash
npx tsc --noEmit 2>&1 | grep "error TS" | wc -l
# Expected: 0
```

### 1.2 Exclude from ESLint

Add an ignore pattern for `src/driver/**` in `eslint.config.js`.

**Verify after:**

```bash
npx eslint src/ tests/ 2>&1 | tail -5
# Expected: 0 problems
```

### 1.3 Run full check suite

```bash
npx tsc --noEmit && npx vitest run && npx eslint src/ tests/ && npm run build
```

All four must pass. If any fail for reasons unrelated to `src/driver/`,
fix before proceeding.

**Write to logbook.** Include the error counts before and after.

## Step 4: Extended Actionability Validation (Runbook Step 9.2)

**Why:** The MVP only tested one actionability scenario (delayed
appearance). Real-world use requires visibility, overlay, scroll, and
disabled-state handling.

### 4.1 Add test scenarios to `demo/toy-app/actionability-test.html`

Add these sections to the existing test page:

1. **Hidden → visible:** A button with `display: none` that becomes
   visible after 1s
2. **Overlay:** A button covered by an absolutely-positioned overlay
   that fades out after 1s
3. **Scroll:** A button below the fold inside a scrollable container
4. **Disabled → enabled:** A button with `disabled` attribute removed
   after 1s

### 4.2 Add test cases to `demo/driver-test.html`

For each new scenario, add a `lite.run()` test that exercises the
actionability behavior:

```javascript
// Example: hidden → visible
await lite.run((cy) => {
  cy.get('#hidden-btn').click();
});
```

### 4.3 Run in browser and verify

All tests should pass — the driver should retry through each
actionability condition automatically.

**Write to logbook.** Record which scenarios pass/fail. If any fail,
note whether the failure is in the driver (retry didn't work) or in the
test setup (wrong HTML structure).

because the driver has its own tsconfig and the bundle builds fine.

### 6.1 Categorize the 51 errors

```bash

  sed 's/.*error //' | sort | uniq -c | sort -rn
```
